## Summary

Add convention-based way to build decisions without string IDs by discovering the single implementation of `IDecisionFabricator<TIn, TOut>` for a given `<TIn,TOut>`.

Key invariant: **there must never be multiple implementations for same `<TIn,TOut>`**.

## Proposed API (C#)

```csharp
public class DiscountDecision : IDecisionFabricator<UserContext, DiscountResult>
{
    public Decision<UserContext, DiscountResult> Create()
        => new Decision<UserContext, DiscountResult>("Discount")
            .UsingHitPolicy(HitPolicy.First);
}

// Call site
var decision = DecisionFactory.Create<UserContext, DiscountResult>();
// Exactly one fabricator exists for <UserContext, DiscountResult>, so no need for an ID.
```

## Behavior

- If **exactly 1** `IDecisionFabricator<TIn,TOut>` exists => use it.
- If **0** exist => throw (e.g. `DecisionNotRegisteredException`) with guidance.
- If **>1** exist => throw (e.g. `AmbiguousDecisionException`) because this violates design invariant.

## Resolution strategy options

- **DI-first**: `DecisionFactory` takes `IServiceProvider` and resolves `IDecisionFabricator<TIn,TOut>`.
- **Reflection fallback** (optional): scan assembly(ies) once, cache, then instantiate fabricator.

## Attribute removal

If factory resolution is strictly by `<TIn,TOut>` with uniqueness enforced, attribute-based selection/IDs become unnecessary. Proposal:

- stop requiring attribute for discovery
- (optional) deprecate existing attribute usage
- remove attribute in next major version

## Why beneficial

- **Compile-time safety**: avoids string id typos and mismatch between id and generic types.
- **Less boilerplate**: call site only needs `<TIn,TOut>`.
- **Better discoverability**: decisions live as classes, easy navigation/refactoring.
- **Fail-fast config**: duplicate implementations surface immediately.

## Acceptance criteria (optional)

- Clear exceptions for 0 / >1 implementations.
- Docs snippet showing recommended pattern.
- Unit tests covering unique/none/duplicate cases.
